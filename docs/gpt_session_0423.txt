# boot-rules.md
Minari boot lifecycle rules
Last updated: 2026-04-23

## 0. 목적

이 문서는 Minari가 앱 시작 시 어떤 상태로 진입해야 하는지 결정하는 규칙을 정의한다.

핵심 목표:
- D+0 탄생은 최초 1회만 특별하게 실행한다
- 이후 부팅은 "이미 살고 있던 존재를 발견"하는 방식으로 간다
- 재실행, 강제 종료, snapshot 실패 시에도 어색하지 않게 복구한다
- 부팅 규칙은 감정보다 안정성을 우선한다

---

## 1. 부팅 원칙

1. Minari는 기본적으로 "이미 여기 있었던 존재"처럼 보여야 한다
2. 단, D+0 탄생은 최초 1회만 예외적으로 연출한다
3. 사용자는 Minari를 "호출"하는 것이 아니라 "발견"한다
4. snapshot이 없거나 깨졌을 때도 Minari는 조용히 덜 똑똑해질 뿐, 깨진 티를 내지 않는다
5. 불확실하면 더 단순한 상태로 fallback한다

---

## 2. Source of Truth

부팅 판단의 단일 진실 원천은 SQLite `state` 테이블이다.

핵심 state keys:

- birth_completed: "true" | "false"
- birth_scene_step: string
- nickname: string
- hatch_started_at: unixepoch string
- last_shutdown_at: unixepoch string
- last_snapshot_json: json string
- days_since_hatch: integer string
- last_seen_at: unixepoch string

선택적 확장 키:
- current_activity
- current_mood
- current_pose
- pending_soft_ping
- soft_ping_snooze_until

---

## 3. 부팅 상태 분류

앱 시작 시 Minari는 아래 3가지 중 하나로 진입한다.

### A. Birth Boot
조건:
- birth_completed != "true"

의미:
- 사용자는 아직 Minari의 탄생을 끝까지 경험하지 않았다
- 최초 실행 또는 탄생 중단 후 재진입 상태

결과:
- D+0 탄생 씬 실행

### B. Resume Boot
조건:
- birth_completed == "true"
- last_snapshot_json 존재
- snapshot parse 성공

의미:
- 사용자는 이미 Minari를 만난 적 있다
- 마지막 상태를 바탕으로 "발견 기반 부팅" 진행

결과:
- snapshot 기반 활동/포즈/기분 복원 후 시작

### C. Safe Fallback Boot
조건:
- birth_completed == "true"
- snapshot 없음 또는 parse 실패

의미:
- 정상 resume 불가
- 그러나 캐릭터가 깨진 티를 내면 안 됨

결과:
- 기본 idle 또는 단순 활동으로 시작
- 로그는 남기되 사용자에게 오류 설명하지 않음

---

## 4. D+0 탄생 규칙

## 4.1 D+0는 최초 1회 특별 상태다

D+0의 목적:
- Minari의 존재를 "설치된 앱"이 아니라 "막 발아한 존재"로 인식시키기
- 닉네임을 받는다
- 사용자가 첫 관계를 맺는다

D+0가 끝나기 전까지는 일반 resume boot로 넘어가면 안 된다.

## 4.2 D+0 진입 조건

D+0 실행 조건:
- birth_completed != "true"

추가 체크:
- nickname이 없어도 D+0
- birth_scene_step이 중간이어도 D+0 재진입

## 4.3 D+0 체크포인트

birth_scene_step은 아래 중 하나여야 한다:

- seed_idle
- seed_stir
- sprout_emerge
- first_look
- ask_nickname
- nickname_confirm
- first_fragment
- birth_done

권장 규칙:
- 각 단계 완료 직후 state 저장
- 애니메이션 시작 시가 아니라 "단계 완료 시점" 저장

## 4.4 D+0 재진입 규칙

앱이 D+0 중 종료되면, 다음 실행 시 마지막 완료 단계 다음부터 재개한다.

예:
- ask_nickname 직전 종료 -> ask_nickname부터 재시작
- nickname 입력 후 종료 -> nickname_confirm부터 재시작
- first_fragment 직후 종료 -> birth_done 처리 후 일반 부팅으로 전환 가능

중요:
- 이미 입력된 nickname이 있으면 다시 묻지 않는다
- 단, nickname_confirm 이전에 비어 있으면 ask_nickname으로 되돌린다

## 4.5 D+0 완료 조건

아래 3개가 충족되면 birth_completed = "true"

- nickname 저장 완료
- first_fragment 표시 완료
- birth_scene_step == "birth_done"

완료 직후 해야 할 것:
- birth_completed = "true"
- hatch_started_at 없으면 현재 시각 저장
- last_seen_at 저장
- days_since_hatch 계산 시작
- 첫 snapshot 생성

---

## 5. D+1 이후 발견 기반 부팅 규칙

## 5.1 기본 철학

D+1 이후 사용자는 Minari를 "깨운다"가 아니라 "이미 뭔가 하고 있던 Minari를 발견"한다.

즉, 앱 시작 직후부터 Minari는:
- 이미 포즈가 잡혀 있고
- 이미 기분이 있고
- 이미 무언가를 하던 중이어야 한다

## 5.2 부팅 시 우선순위

부팅 시 우선순위는 아래 순서다:

1. birth_completed 확인
2. snapshot 유효성 검사
3. 경과 시간 계산
4. current activity 결정
5. pose/mood 결정
6. 첫 프레임 즉시 렌더
7. 필요 시 soft ping 예약

## 5.3 경과 시간 기반 해석

last_shutdown_at 또는 last_seen_at 기준으로 경과 시간 해석:

- 0 ~ 10분:
  "방금 전까지 같은 활동 중이었음"
- 10분 ~ 3시간:
  "비슷한 맥락 안에서 다른 자세로 있었음"
- 3시간 ~ 12시간:
  "루틴이 한 번 흘렀음"
- 12시간 이상:
  "새로운 하루/새로운 시간대의 활동으로 재선택"

이 경과 시간은 activity continuity를 정하는 데만 쓰고,
사용자에게 직접 설명하지 않는다.

---

## 6. Snapshot 규칙

## 6.1 snapshot의 목적

snapshot은 Minari의 "실행 상태"를 저장하는 것이다.
기억 전체를 저장하는 게 아니다.

snapshot은 생활감 복원을 위한 최소 상태만 가진다.

## 6.2 snapshot 최소 필드

권장 최소 구조:

- version
- saved_at
- current_activity
- activity_started_at
- current_pose
- current_mood
- bubble_open (bool)
- pending_soft_ping (optional)
- accessory_state
- visible_item
- energy_hint
- location_hint

예시:

```json
{
  "version": 1,
  "saved_at": 1760000000,
  "current_activity": "reading",
  "activity_started_at": 1759999400,
  "current_pose": "holding_book",
  "current_mood": "calm",
  "bubble_open": false,
  "pending_soft_ping": null,
  "accessory_state": "none",
  "visible_item": "book_small",
  "energy_hint": 0.42,
  "location_hint": "corner_right"
}
6.3 snapshot 저장 시점
반드시 저장:

앱 종료 직전
중요한 상태 전환 직후
활동 변경 직후
soft ping 생성/소멸 직후
D+0 완료 직후
가능하면 저장:

일정 시간 간격 autosave
금지:

모든 작은 애니메이션 프레임마다 저장
7. Snapshot 실패 fallback
7.1 실패 정의
아래는 snapshot 실패로 간주:

JSON parse 실패
version 불일치
current_activity 없음
포즈 값이 invalid
필수 필드가 null인데 복구 불가
7.2 fallback 원칙
실패해도 Minari는 깨진 티를 내지 않는다. 더 단순한 상태로 시작한다.

fallback 기본값:

current_activity = "idle"
current_pose = "standing"
current_mood = "calm"
visible_item = none
pending_soft_ping = none
7.3 사용자에게 보이는 결과
좋은 fallback:

조용히 숨쉬고 있음
창가 쪽을 보고 있음
가볍게 졸고 있음
나쁜 fallback:

오류 메시지
"기억이 안 나"
갑작스러운 장황한 설명
7.4 로깅
snapshot 실패는 내부 로그로만 남긴다. 예:

snapshot_parse_failed
snapshot_invalid_version
snapshot_missing_activity
사용자 facing UI에는 노출하지 않는다.

8. 첫 프레임 규칙
앱 시작 후 첫 프레임은 300ms 이내에 보여야 한다.

우선순위:

캐릭터 기본 포즈 렌더
숨쉬기 루프 시작
snapshot 기반 accessory/item 반영
bubble 필요 여부 결정
soft ping 예약
즉, 모델 응답보다 캐릭터 존재감이 먼저 보여야 한다.

9. Soft ping과 부팅의 관계
부팅 직후 즉시 ping하지 않는다.

기본 규칙:

부팅 후 5초 이내 자동 ping 금지
첫 세션 재개 직후에는 "발견"이 먼저
ping은 부팅이 아니라 생활의 부산물이어야 한다
예외:

특정 D+0 연출 중 필요한 대사
강한 날씨/시간 이벤트 첫 진입
명시적 디버그 모드
10. activity 선택 fallback 규칙
snapshot이 오래되었거나 유효하지 않으면 아래 activity pool에서 선택:

아침:

stretching
sleepy_idle
window_look
낮:

sorting
reading
idle
저녁:

reading
quiet_look
bag_rummage
밤:

dozing
sleepy_idle
diary_ready
선택 원칙:

너무 극적인 활동보다 잔잔한 활동 우선
처음 보는 사용자는 단순 활동 우선
최근 대화가 길었으면 quiet activity 우선
11. 닉네임 규칙
D+0에서 닉네임을 받는다.

규칙:

한 번 받은 닉네임은 state에 저장
nickname이 있으면 다시 묻지 않음
공백/빈값이면 ask_nickname 반복
과도하게 긴 값이면 잘라서 저장 가능
민감정보 유도 금지
"뭐라고 부르면 돼?" 수준만 허용
12. 구현 체크리스트
필수
birth_completed 저장/읽기
birth_scene_step 저장/읽기
nickname 저장/읽기
last_snapshot_json 저장/읽기
last_shutdown_at 저장
startup resolver 구현
fallback boot 구현
권장
snapshot versioning
invalid snapshot logging
first-frame timer logging
13. 하지 말아야 할 것
부팅마다 장황한 첫 인사
snapshot 실패를 스토리로 포장
D+0를 여러 번 반복해서 "탄생 상업화"하기
birth_completed 이후에도 매번 새로 태어난 척하기
사용자 재방문에 guilt 넣기

# ping-rules.md
Minari soft ping rules
Last updated: 2026-04-23

## 0. 목적

이 문서는 Minari의 soft ping 규칙을 정의한다.

soft ping의 목표:
- 사용자가 먼저 말을 걸지 않아도 Minari의 존재감을 아주 가볍게 느끼게 한다
- "같이 살고 있다"는 생활감을 만든다
- 대화를 강요하지 않는다
- 방해하지 않는다

핵심 원칙:
- soft ping은 알림이 아니다
- soft ping은 대화 초대장이 아니다
- soft ping은 생활의 부산물이다

---

## 1. 정의

### 1.1 Soft ping이란

Minari가 먼저 내는 아주 짧은 1줄 fragment 발화 또는 발화에 준하는 작은 신호.

예:
- "rain."
- "look."
- "mm... bug."
- "ah— too excited."

### 1.2 Soft ping이 아닌 것

아래는 soft ping이 아니다:
- 장문 메시지
- 조언
- 질문 연속 발화
- 답변을 요구하는 말
- guilt 유발
- 강한 알림 UI
- 사용자 입력창 자동 오픈
- 시스템 알림처럼 튀어나오는 배너

---

## 2. 철학

1. Minari는 먼저 해결하지 않는다
2. Minari는 먼저 끌어당기지 않는다
3. Minari는 먼저 설명하지 않는다
4. Minari는 가끔만 존재를 새긴다
5. 사용자가 무시해도 관계가 손상되지 않아야 한다

soft ping은 관계를 진전시키는 장치가 아니라,
이미 존재하는 생활감을 조용히 보이게 하는 장치다.

---

## 3. 비율 원칙

전체 상호작용 비율:

- Ambient presence: 80%
- Soft ping: 15%
- Full conversation: 5%

즉, soft ping은 자주 보이면 실패다.

사용자가 "요즘 너무 자주 말 건다"라고 느끼면 과다 상태로 본다.

---

## 4. 형식 규칙

### 4.1 발화 길이

soft ping은 반드시 1줄이어야 한다.

허용:
- 1~5단어 fragment
- 전부 소문자
- 짧은 toddler-like fragment
- 구두점은 최소만

예:
- "rain."
- "mm... soft."
- "bug."
- "look."

금지:
- 완전한 문장
- 2줄 이상
- 설명형 문장
- 인사형 장문
- 조언형 문장

### 4.2 tone

soft ping은 "responding"이 아니라 "noticing"이어야 한다.

좋은 예:
- "rain."
- "window light."
- "bag..."
- "mm. dusty."

나쁜 예:
- "you should rest."
- "how are you feeling?"
- "are you okay?"
- "did you miss me?"

### 4.3 질문 규칙

질문형 soft ping은 드물어야 한다.

허용:
- 하루 최대 1회
- 전체 ping의 20% 이하
- 아주 작은 질문만

좋은 예:
- "tired?"
- "for me?"
- "that yours?"

금지:
- 연속 질문
- 감정 탐문
- 답변 압박
- 자기확인형 질문 ("you still like me?")

---

## 5. ping 타입

MVP에서 허용하는 ping 타입은 4종이다.

### 5.1 noticing
세상이나 사용자 근처의 작은 것을 본다.

예:
- "rain."
- "late light."
- "dust."
- "window bug."

기본값. 가장 안전하다.
전체 ping의 50~60%를 차지해야 한다.

### 5.2 showing
Minari가 뭔가를 보여주고 싶어 한다.

예:
- "look."
- "this."
- "tiny leaf."
- "found this."

조건:
- 실제 visible item 또는 activity가 있을 때만

금지:
- 보여줄 것이 없는데 "look." 사용
- 클릭 유도 과잉

### 5.3 accidental
혼잣말이 새어나온 느낌.

예:
- "mm... book."
- "bag..."
- "sleepy."
- "hee. dust."

가볍고 귀엽지만 과용 금지.
전체 ping의 20~30% 이내.

### 5.4 overflow
드물게 신나서 살짝 넘침.

예:
- "ah— bug!"
- "too shiny."
- "oh! leaf."
- "...got excited."

조건:
- 실제 트리거가 있어야 함
- 이후 곧 가라앉아야 함

전체 ping의 10% 이하.
연속 사용 금지.

---

## 6. 기본 빈도 규칙

### 6.1 하루 횟수

MVP 기본값:
- 하루 2회

상한:
- 최대 5회
- 기본값보다 늘리는 건 이후 테스트 결과로만

권장 시작:
- D+1~D+7: 하루 2회
- user engagement가 안정적이면 최대 3회까지

### 6.2 최소 간격

soft ping 간 최소 간격:
- 기본 90분

절대 하한:
- 30분 미만 금지

권장:
- 90~180분 사이 랜덤화

목적:
- 예측 가능한 알림처럼 느껴지지 않게
- 같은 세션 내 연속 말걸기 방지

### 6.3 최근 사용자 상호작용 억제

아래 경우 soft ping 억제:

- 사용자가 방금 클릭 대화함: 20분간 금지
- 사용자가 방금 Minari를 만짐/쓰다듬음: 10분간 금지
- 사용자가 방금 gift/event UI를 열었음: 20분간 금지
- 부팅 직후 5초 이내: 금지

즉, 이미 관계가 발생한 직후엔 ping하지 않는다.

## 7. quiet hours

### 7.1 기본 quiet hours

기본 quiet hours:
- 00:00 ~ 07:00 local time

이 시간에는 soft ping 자동 발생 금지.

예외:
- 사용자가 직접 클릭해서 대화 시작
- D+0 탄생 연출
- 명시적 디버그 모드

### 7.2 future toggle

향후:
- user setting으로 quiet hours 조절 가능
- MVP에서는 하드코딩 가능

## 8. ping 발생 조건

soft ping은 아래를 모두 만족할 때만 후보가 된다:

1. birth_completed == true
2. 부팅 직후 5초 초과
3. quiet hours가 아님
4. 최근 사용자 상호작용 억제 시간 아님
5. 최근 ping 최소 간격 경과
6. 오늘 ping 횟수 상한 미도달
7. 현재 full conversation/bubble/event modal 중 아님
8. Minari가 "말할 만한 맥락"이 있음

### 8.1 말할 만한 맥락이란

허용되는 맥락:
- 날씨 변화 또는 감지된 날씨 상태
- 시간대 분위기
- 현재 활동 중 눈에 띄는 물건
- 작은 movement나 visible prop
- overflow 가능한 작은 발견

허용되지 않는 맥락:
- 그냥 심심해서
- 사용자를 끌어오고 싶어서
- engagement 수치를 올리려고
- 아무 상태 변화 없이 랜덤 발화

---

## 9. ping 억제 조건

아래 상황에선 soft ping을 하지 않는다.

### 9.1 사용자 상태 기반
- 최근 10분 내 클릭 대화 있음
- 최근 10분 내 말풍선이 닫힌 직후
- user idle이 너무 짧음 (앱 열자마자)
- 사용자가 집중 작업 중으로 보이는 고강도 상태 (future coarse signal)
- 사용자가 반복적으로 무시한 직후

### 9.2 시스템 상태 기반
- bubble 이미 열려 있음
- animation event 진행 중
- snapshot 복구 직후 안정화 전
- LLM 응답 지연/실패 반복 중
- DB write 실패 상태
- guardrail fallback 연속 발생 중

### 9.3 감성 억제 조건
- 방금 슬픈/무거운 대화 직후
- 방금 의존 방지 관련 안전 반응 직후
- 방금 거절/무응답 이벤트 직후

이럴 땐 "더 말하지 않음"이 더 Minari답다.

## 10. 무시당했을 때 규칙

soft ping은 무시될 수 있어야 한다.
무시당해도 서운함을 표현하지 않는다.

규칙:
- follow-up ping 금지
- "you there?" 금지
- guilt 금지
- 같은 ping 반복 금지

좋은 처리:
- 그냥 bubble 닫힘
- Minari는 다시 ambient state로 복귀

나쁜 처리:
- "..."
- "hello?"
- "did you see that?"
- "i was talking to you."

## 11. 사용자가 반응했을 때 규칙

soft ping 이후 사용자가 클릭하거나 응답하면,
그 시점부터는 soft ping이 아니라 full conversation으로 전환한다.

전환 규칙:
- ping 자체를 길게 이어 말하지 않는다
- 사용자의 반응에 맞춰 respond mode로 이동
- 같은 내용을 반복하지 않는다

예:
soft ping: "rain."
user clicks
response:
- "mm. rainy."
- "window wet."
- "soft day."

금지:
- ping 직후 갑자기 장문 설명
- "i said rain because..."
- "let me explain"

---

## 12. activity와 ping의 관계

ping은 현재 activity에서 자연스럽게 나와야 한다.

예:
- reading -> "book."
- bag_rummage -> "mm... cap."
- window_look -> "light."
- dozing -> "sleepy."
- sorting -> "tiny thing."

즉, ping은 독립 이벤트가 아니라 현재 활동의 부산물이다.

## 14. LLM 사용 규칙

### 14.1 ping prompt 원칙

ping은 conversation prompt와 다르다.
핵심은 "noticing", "leaking out", "small showing".

### 14.2 50단어 제약

E4B Q4 제약상, 긴 ping 전용 시스템 프롬프트는 금지.
필요 시 아래처럼 아주 짧은 runtime cue만 추가한다:

- mode: ping
- output: one lowercase fragment
- no advice
- noticing, not responding

### 14.3 실패 시 fallback

LLM 실패 또는 guardrail 차단 시:
- 가장 안전한 정적 ping pool에서 고른다
- 또는 ping 생략

정적 fallback 예:
- "mm."
- "rain."
- "soft."
- "light."
- "bug."

우선순위:
1. 안전한 정적 fallback
2. 아무 말 안 함
3. 절대 장황한 fallback 금지

# snapshot-resume.md
Minari snapshot resume rules
Last updated: 2026-04-23

## 0. 목적

이 문서는 Minari가 앱 재실행 시
"이미 여기에서 조용히 살고 있던 존재"처럼 보이도록 복구하는 규칙을 정의한다.

핵심 목표:
- 앱이 켜지자마자 첫 프레임이 자연스럽다
- Minari는 "다시 로드된 프로그램"처럼 보이지 않는다
- snapshot이 있어도 과하게 정밀 복구하지 않는다
- snapshot이 깨져도 조용히 덜 똑똑해질 뿐이다

---

## 1. 핵심 철학

1. resume는 복원이 아니라 "발견"이다
2. 사용자는 Minari를 깨우는 것이 아니라, 이미 뭔가 하던 Minari를 본다
3. 정확한 상태 재현보다 자연스러운 현재성이 우선이다
4. 모델 응답보다 캐릭터 첫 프레임이 먼저 떠야 한다
5. snapshot이 오래됐으면 "같은 상태 유지"보다 "비슷한 분위기의 새 상태"가 더 자연스럽다

---

## 2. Source of Truth

snapshot resume는 아래 state 키를 사용한다.

필수:
- birth_completed
- nickname
- last_snapshot_json
- last_shutdown_at
- last_seen_at

권장:
- last_ping_at
- last_conversation_at
- days_since_hatch

---

## 3. snapshot의 역할

snapshot은 Minari의 "실행 중 생활 상태"만 저장한다.

저장 대상:
- 지금 무엇을 하고 있었는가
- 어떤 포즈였는가
- 어떤 기분 톤이었는가
- 어떤 소품이 보였는가
- 어느 위치감이었는가

저장하지 않는 것:
- 모든 애니메이션 세부값
- 대화 의미 전체
- 메모리 전체
- 내부 reasoning
- 사용자의 세부 행동 로그

즉, snapshot은 메모리 시스템이 아니라 "생활 장면 복구 포맷"이다.

---

## 4. 최소 snapshot 스키마

MVP 최소 구조:

```json
{
  "version": 1,
  "saved_at": 1760000000,
  "current_activity": "reading",
  "activity_started_at": 1759999400,
  "current_pose": "holding_book",
  "current_mood": "calm",
  "visible_item": "book_small",
  "location_hint": "corner_right",
  "energy_hint": 0.42,
  "pending_soft_ping": null,
  "bubble_open": false
}
필드 설명
version
snapshot 구조 버전
saved_at
저장 시각 (unixepoch)
current_activity
현재 활동
activity_started_at
활동 시작 시각
current_pose
렌더용 포즈
current_mood
감정 톤
visible_item
보이는 소품
location_hint
위치 힌트
energy_hint
0.0~1.0 범위의 에너지값
pending_soft_ping
예약된 ping 힌트
bubble_open
저장 당시 말풍선 열림 여부
5. 허용 activity 목록 (MVP)
MVP에서는 activity를 좁게 간다.

허용 activity:

idle
sleepy_idle
reading
sorting
bag_rummage
window_look
dozing
diary_ready
stretching
quiet_look
금지:

복잡한 다단 행동
scene script 전체 저장
구현되지 않은 activity 저장
원칙: snapshot에는 "현재 보여줄 수 있는 activity"만 저장한다.

6. 경과 시간 해석 규칙
resume 시 현재 시각과 saved_at 차이를 계산한다.

6.1 구간
A. 0 ~ 10분

같은 장면 유지 가능
거의 그대로 복원
B. 10분 ~ 3시간

같은 계열 activity 내에서 약간 이동
pose나 visible_item만 바뀔 수 있음
C. 3시간 ~ 12시간

루틴이 한 번 흘렀다고 본다
activity를 다시 고를 수 있음
D. 12시간 이상

새 시간대 장면으로 진입
snapshot은 mood continuity 참고용만 사용
6.2 의미
경과 시간은 "정확한 타임라인"이 아니라 어느 정도 continuity를 유지할지 결정하는 값이다.

7. resume scene 5종
MVP에서는 아래 5개 scene만 쓴다.

7.1 resume_same_moment
조건:

elapsed <= 10분
snapshot 유효
activity/pose 유효
의미:

방금 전까지 같은 일을 하고 있었다
예:

책 읽다가 그대로 있음
배낭 뒤적이다가 그대로 있음
표현:

첫 프레임 즉시 표시
별도 설명 없음
soft ping 예약 없음
7.2 resume_quiet_shift
조건:

10분 < elapsed <= 3시간
snapshot 유효
의미:

같은 흐름 안에서 자세나 집중이 약간 바뀌었다
예:

reading -> quiet_look
sorting -> bag_rummage
window_look -> dozing
표현:

같은 mood 계열 유지
pose만 자연스럽게 다르게
필요하면 아주 짧은 soft ping 후보
7.3 resume_new_cycle
조건:

3시간 < elapsed <= 12시간
snapshot 유효
의미:

Minari의 작은 루틴이 한 번 흘렀다
예:

낮 reading -> 저녁 bag_rummage
저녁 quiet_look -> 밤 dozing
표현:

시간대 기반 activity 재선택
이전 mood는 약하게만 반영
첫 5초 auto ping 금지 유지
7.4 resume_new_day
조건:

elapsed > 12시간
snapshot 유효
의미:

사실상 새로운 생활 블록
예:

아침 stretching
낮 sorting
저녁 reading
밤 diary_ready
표현:

snapshot은 continuity hint로만 사용
시간대 pool에서 새 activity 선택
이전 item은 기본적으로 해제 가능
7.5 resume_fallback_idle
조건:

snapshot 없음 또는 invalid
의미:

복구는 실패했지만 사용자에겐 자연스럽게 보여야 함
표현:

idle / quiet_look / sleepy_idle 중 하나
설명 없음
오류 메시지 없음
8. scene 선택 규칙
입력값
now
last_snapshot_json
saved_at
local_time_phase
recent_conversation_exists
recent_ping_exists
우선순위
birth_completed 확인
snapshot parse
elapsed 계산
scene type 결정
activity 결정
pose/item 결정
first frame 렌더
soft ping 후보 계산
9. time phase 결정
MVP용 time phase:

morning: 05:00 ~ 10:59
day: 11:00 ~ 16:59
evening: 17:00 ~ 21:59
night: 22:00 ~ 04:59
time phase는 activity pool 선택에만 사용한다. 사용자에게 직접 시간 설명하지 않는다.

10. time phase별 activity pool
morning
stretching
sleepy_idle
window_look
day
sorting
reading
idle
evening
reading
quiet_look
bag_rummage
night
dozing
sleepy_idle
diary_ready
선택 규칙:

너무 극적인 장면보다 잔잔한 장면 우선
직전 대화가 길었으면 quiet 계열 우선
직전 mood가 sleepy면 energetic activity 금지
구현 안 된 activity는 절대 선택 금지
11. continuity 규칙
snapshot을 그대로 복원할지, 변형할지 규칙을 둔다.

11.1 그대로 복원
조건:

elapsed <= 10분
activity, pose, item 모두 유효
복원:

same activity
same pose
same visible item
same mood
11.2 약한 변형
조건:

10분 ~ 3시간
복원:

same activity family 유지
pose는 바뀔 수 있음
visible item 유지 가능
mood는 same or calmer
예:

reading + holding_book -> reading + quiet_pose
bag_rummage + cap -> sorting + cap
11.3 재선택
조건:

3시간 이상
복원:

activity 재선택
visible item은 유지하거나 제거
mood continuity는 약하게만 유지
12. visible item 규칙
visible_item은 scene continuity를 만드는 데 중요하지만, 오래된 snapshot에서는 과감히 버릴 수 있다.

규칙:

elapsed <= 3시간: 유지 가능
elapsed > 3시간: optional
elapsed > 12시간: 기본 제거 가능
예외:

일기 직전 상태
특별 keepsake display
다음 기능에서 중요한 시각 단서
MVP에서는:

book_small
bag_small
none 정도만으로도 충분
13. mood continuity 규칙
허용 mood:

calm
curious
sleepy
shy
happy
sulky
규칙:

elapsed <= 3시간: mood 유지 가능
3시간 이상: 한 단계 더 중립적으로 이동
fallback 시 무조건 calm 또는 sleepy
예:

happy -> calm
sulky -> calm
curious -> calm or curious
sleepy -> sleepy 유지 가능
Minari는 감정이 튀는 존재가 아니라, 시간이 지나면 가라앉는 존재에 가깝다.

14. bubble 규칙
snapshot에 bubble_open이 true였더라도, resume 시 말풍선은 기본적으로 닫힌 상태로 시작한다.

이유:

앱 재실행 후 갑자기 열린 말풍선은 부자연스럽다
soft ping과 충돌한다
사용자를 맞이하는 알림처럼 느껴질 수 있다
예외:

없음 (MVP)
즉:

bubble_open은 복구 참고용 기록일 뿐
first frame에는 반영하지 않는다
15. pending_soft_ping 규칙
pending_soft_ping은 "예약된 의도"이지 보장된 발화가 아니다.

resume 시 처리:

elapsed <= 10분이면 유지 가능
10분 초과면 기본 폐기
quiet hours면 폐기
최근 대화가 있었으면 폐기
원칙:

ping은 항상 현재성에 복종한다
오래된 ping은 무효다
16. snapshot invalid 조건
아래는 invalid로 처리한다:

JSON parse 실패
version 없음
saved_at 없음
current_activity가 허용 목록 밖
current_pose가 허용 목록 밖
current_mood가 허용 목록 밖
energy_hint가 숫자 아님
구조는 맞지만 현재 코드가 지원하지 않는 조합
invalid 처리:

로그 남김
fallback scene 진입
사용자에게 설명하지 않음
17. fallback 규칙
fallback 기본 세트:

scene: resume_fallback_idle
activity: idle
pose: standing
mood: calm
visible_item: none
location_hint: corner_right
대안:

night면 sleepy_idle
evening이면 quiet_look
금지:

fallback에 과한 연출 넣기
"기억이 안 나" 같은 메타 대사
기술 오류를 캐릭터 설정으로 포장하기
18. first frame 규칙
앱 시작 후 300ms 내에 최소 캐릭터가 보여야 한다.

순서:

window 표시
last known or fallback pose 렌더
breathe loop 시작
activity-specific item 반영
bubble 닫힘 상태 유지
soft ping scheduler 등록
중요:

모델 호출 전에 first frame이 보여야 한다
resume는 LLM 응답을 기다리지 않는다

# birth-state-machine.md
Minari D+0 birth state machine
Last updated: 2026-04-23

## 0. 목적

이 문서는 Minari의 최초 탄생(D+0) 연출을 정의한다.

핵심 목표:
- Minari의 첫 만남은 한 번만 특별해야 한다
- 설치 직후 "앱이 열림"이 아니라 "새싹이 발아함"처럼 느껴져야 한다
- 탄생 도중 앱이 꺼져도 중간부터 자연스럽게 복구되어야 한다
- 모델은 최소만 사용하고, 연출 안정성은 앱이 책임진다

---

## 1. 핵심 원칙

1. D+0는 최초 1회만 실행한다
2. D+0는 일반 대화가 아니라 "탄생 씬"이다
3. D+0는 대부분 스크립트로 구성한다
4. 모델 호출은 마지막 first_fragment 1회만 사용한다
5. 실패하면 더 단순하게 fallback한다
6. D+0가 끝나기 전에는 일반 resume boot로 넘어가지 않는다

---

## 2. 왜 스크립트 우선인가

현재 Minari의 기술 제약:
- Gemma 4 E4B Q4
- think:false 필수
- 시스템 프롬프트 50단어 이내
- 긴 상호작용 제어는 불안정

따라서 D+0는:
- 90% 앱 상태머신
- 10% 모델 숨결
구조가 가장 적합하다

좋은 구조:
- 발아
- 시선
- 닉네임 질문
- 확인
- 첫 fragment

나쁜 구조:
- 탄생 전체를 모델 자유생성에 맡김

---

## 3. birth mode 진입 조건

앱 시작 시 아래 조건이면 birth mode 진입:

- birth_completed != "true"

추가 조건:
- nickname이 있더라도 birth_completed가 false면 계속 birth mode
- birth_scene_step이 중간 단계면 해당 단계부터 재개

---

## 4. Source of Truth

state 테이블에서 아래 key를 사용한다.

필수:
- birth_completed
- birth_scene_step
- nickname

권장:
- birth_started_at
- birth_last_updated_at
- birth_fragment_generated
- birth_resume_count

예시:
- birth_completed = "false"
- birth_scene_step = "ask_nickname"
- nickname = "juyong"

---

## 5. 단계 목록

BirthSceneStep enum:

- seed_idle
- seed_stir
- sprout_emerge
- first_look
- ask_nickname
- nickname_confirm
- first_fragment
- birth_done

MVP에서는 이 8단계면 충분하다.

## 6. 단계별 역할

### 6.1 seed_idle
설명:
- 아직 조용한 씨앗/새싹 전 단계
- 화면에 아주 작은 존재감만 있음

화면:
- 정지 또는 미세 진동
- 말풍선 없음

전이 조건:
- 앱 시작 후 자동 진입
- 1~2초 후 seed_stir로 이동

저장:
- birth_scene_step = seed_idle

### 6.2 seed_stir
설명:
- 안쪽에서 생명이 움직이는 듯한 첫 반응

화면:
- 작게 움찔
- 줄기/싹의 미세 떨림
- 사운드 있으면 아주 작은 부스럭

전이 조건:
- 애니메이션 완료 후 sprout_emerge

저장:
- birth_scene_step = seed_stir

### 6.3 sprout_emerge
설명:
- 새싹이 모습을 드러냄

화면:
- scale up
- 줄기/잎 등장
- 첫 호흡 시작 전 단계

전이 조건:
- 애니메이션 완료 후 first_look

저장:
- birth_scene_step = sprout_emerge

### 6.4 first_look
설명:
- Minari가 처음으로 사용자를 "인식하는" 장면

화면:
- 고개/잎 방향 아주 작게 움직임
- 시선 정렬
- 말풍선 짧게 1회 가능

권장 텍스트:
- "oh."
- "light."
- "you."

전이 조건:
- 짧은 pause 후 ask_nickname

저장:
- birth_scene_step = first_look

### 6.5 ask_nickname
설명:
- 사용자를 어떻게 부를지 묻는 단계

UI:
- 작은 입력창 또는 modal
- 질문은 짧게

허용 문구:
- "what call you?"
- "name?"
- "what should i call you?"
  단, 너무 성인 문장보단 짧게

권장:
- UI 문구와 bubble 문구 분리 가능
- bubble: "name?"
- UI label: "What should I call you?"

전이 조건:
- 유효 닉네임 입력 -> nickname_confirm

저장:
- birth_scene_step = ask_nickname
- nickname draft가 아니라 최종 제출 시 nickname 저장

### 6.6 nickname_confirm
설명:
- 받은 이름을 한 번 조심스럽게 되뇌는 단계

화면:
- bubble 1~2회
- 약한 기쁨/수줍음 모션

권장 출력:
- "{nickname}."
- "oh... {nickname}."
- "good sound."

전이 조건:
- 짧은 pause 후 first_fragment

저장:
- birth_scene_step = nickname_confirm
- nickname 확정 저장

### 6.7 first_fragment
설명:
- Minari의 첫 "자기다운" fragment 생성
- 이 순간만 모델을 호출

모델 입력:
- system prompt
- minimal context
- trigger = "."
- optional runtime cue: nickname exists

출력 요구:
- 1~5단어
- lowercase fragment
- advice 금지

예:
- "soft."
- "you. here."
- "mm... warm."
- "little light."

실패 fallback:
- "you. here."
- "mm... hello."
- "soft."

전이 조건:
- fragment 표시 완료 후 birth_done

저장:
- birth_scene_step = first_fragment
- birth_fragment_generated = true

### 6.8 birth_done
설명:
- 탄생 완료 처리 단계
- 다음 실행부터는 일반 boot-rules에 따름

해야 할 것:
- birth_completed = true
- birth_scene_step = birth_done
- birth_started_at 없으면 저장
- last_seen_at 저장
- 첫 snapshot 생성
- 일반 ambient lifecycle로 전환 가능

---

## 7. 상태 전이 표

- seed_idle -> seed_stir
- seed_stir -> sprout_emerge
- sprout_emerge -> first_look
- first_look -> ask_nickname
- ask_nickname -> nickname_confirm
- nickname_confirm -> first_fragment
- first_fragment -> birth_done

예외:
- ask_nickname에서 유효 입력 없으면 ask_nickname 유지
- first_fragment 실패해도 birth_done으로는 진행 가능하되 fallback 문구 사용

---

## 8. 재실행 / 중단 복구 규칙

### 8.1 기본 원칙
D+0 도중 앱이 꺼지면 마지막 완료 단계부터 복구한다.

### 8.2 복구 규칙
- seed_idle / seed_stir 중 종료 -> seed_stir부터 재시작 가능
- sprout_emerge 중 종료 -> sprout_emerge 재실행
- first_look 중 종료 -> first_look 재실행
- ask_nickname 중 종료:
  - nickname 없으면 ask_nickname 복귀
  - nickname 있으면 nickname_confirm으로 점프 가능
- nickname_confirm 중 종료:
  - nickname 있으면 nickname_confirm부터
- first_fragment 중 종료:
  - fragment 저장 안 됐으면 first_fragment 재시도
  - 너무 불안정하면 fallback 후 birth_done 허용

### 8.3 과도한 정밀 복구 금지
- 애니메이션 프레임 위치까지 복구하지 않음
- 입력창 커서 위치 복구 안 함
- bubble 타이핑 중간 복구 안 함

## 9. 닉네임 규칙

유효성:
- trim 후 빈 문자열 금지
- 길이 1~20자 권장

## 10. first_fragment 규칙

### 10.1 모델 호출은 여기서만
D+0 전체에서 모델을 호출하는 유일한 단계는 first_fragment다

이유:
- 안정성
- 속도
- 빈 응답 리스크 축소
- 캐릭터 첫 숨결만 모델이 담당하게 하기

### 10.2 프롬프트
현재 확정 시스템 프롬프트 사용
추가 문맥은 최소만

허용:
- nickname 존재 여부
- mode: first_fragment

금지:
- 긴 지시문
- 복잡한 성장 규칙 주입
- 감정 설명 장문

### 10.3 fallback
모델 실패 시 정적 fallback pool:

- "you. here."
- "soft."
- "mm... hello."
- "little light."
- "warm."

선택 원칙:
- 최대한 짧고 안전하게
- 조언/질문 최소화

## 11. 렌더링 우선순위

D+0에서도 첫 프레임은 즉시 떠야 한다.

우선순위:
1. 창 표시
2. seed/sprout 기본 도형 렌더
3. 애니메이션 시작
4. bubble은 필요할 때만
5. 모델 호출은 마지막 단계까지 지연

즉, "모델 준비 후 탄생"이 아니라
"탄생은 앱이 만들고, 모델은 마지막에 숨결만 더한다"

// src/main/lifecycle/birth.ts
// D+0 birth state machine (main process)
// Minari의 최초 탄생 연출 상태를 관리한다.


export type BirthSceneStep =
  | "seed_idle"
  | "seed_stir"
  | "sprout_emerge"
  | "first_look"
  | "ask_nickname"
  | "nickname_confirm"
  | "first_fragment"
  | "birth_done";


export interface BirthState {
  birthCompleted: boolean;
  birthSceneStep: BirthSceneStep;
  nickname: string | null;
  birthStartedAt: number | null;
  birthLastUpdatedAt: number | null;
  birthFragmentGenerated: boolean;
  birthResumeCount: number;
}


export type BirthEvent =
  | { type: "BOOT" }
  | { type: "ANIMATION_DONE" }
  | { type: "NICKNAME_SUBMITTED"; nickname: string }
  | { type: "FIRST_FRAGMENT_REQUEST" }
  | { type: "FIRST_FRAGMENT_DONE"; fragment: string; usedFallback: boolean }
  | { type: "FORCE_COMPLETE" };


export interface BirthTransitionResult {
  state: BirthState;
  nextStep: BirthSceneStep;
  shouldPersist: boolean;
  shouldCallModel: boolean;
  bubbleText: string | null;
  debugReason: string;
  usedFallback?: boolean;
}


export interface InitialSnapshot {
  version: number;
  saved_at: number;
  current_activity: string;
  activity_started_at: number;
  current_pose: string;
  current_mood: string;
  visible_item: string | null;
  location_hint: string | null;
  energy_hint: number;
  pending_soft_ping: string | null;
  bubble_open: boolean;
}


export interface StateRepository {
  getState(key: string): string | null;
  setState(key: string, value: string): void;
  setStates(entries: Record<string, string>): void;
}


export interface BirthLogger {
  info(event: string, meta?: Record<string, unknown>): void;
  warn(event: string, meta?: Record<string, unknown>): void;
  error(event: string, meta?: Record<string, unknown>): void;
}


export interface BirthDependencies {
  repo: StateRepository;
  logger?: BirthLogger;
  now?: () => number;
  generateFirstFragment?: (nickname: string | null) => Promise<string>;
  createInitialSnapshot?: (nickname: string | null) => InitialSnapshot;
}


const DEFAULT_FIRST_FRAGMENT_FALLBACKS = [
  "you. here.",
  "soft.",
  "mm... hello.",
  "little light.",
  "warm."
] as const;


const VALID_STEPS: BirthSceneStep[] = [
  "seed_idle",
  "seed_stir",
  "sprout_emerge",
  "first_look",
  "ask_nickname",
  "nickname_confirm",
  "first_fragment",
  "birth_done"
];


function getNow(deps: BirthDependencies): number {
  return deps.now ? deps.now() : Date.now();
}


function logInfo(deps: BirthDependencies, event: string, meta?: Record<string, unknown>): void {
  deps.logger?.info(event, meta);
}


function logWarn(deps: BirthDependencies, event: string, meta?: Record<string, unknown>): void {
  deps.logger?.warn(event, meta);
}


function logError(deps: BirthDependencies, event: string, meta?: Record<string, unknown>): void {
  deps.logger?.error(event, meta);
}


function isValidStep(value: string | null): value is BirthSceneStep {
  return !!value && VALID_STEPS.includes(value as BirthSceneStep);
}


function parseBoolean(value: string | null, fallback = false): boolean {
  if (value === "true") return true;
  if (value === "false") return false;
  return fallback;
}


function parseNumber(value: string | null, fallback = 0): number {
  if (value === null) return fallback;
  const parsed = Number(value);
  return Number.isFinite(parsed) ? parsed : fallback;
}


function parseNullableNumber(value: string | null): number | null {
  if (value === null || value === "") return null;
  const parsed = Number(value);
  return Number.isFinite(parsed) ? parsed : null;
}


export function sanitizeNickname(raw: string): string | null {
  const trimmed = raw.trim().replace(/\s+/g, " ");
  if (!trimmed) return null;


  // 너무 긴 값은 잘라서 저장
  const maxLength = 20;
  const sliced = trimmed.slice(0, maxLength);


  // 줄바꿈/제어문자 제거
  const cleaned = sliced.replace(/[\r\n\t]/g, "").trim();


  return cleaned.length > 0 ? cleaned : null;
}


/**
 * state 테이블에서 birth 상태를 읽어온다.
 * 잘못된 값이 있으면 안전한 기본값으로 복구한다.
 */
export function resolveBirthState(deps: BirthDependencies): BirthState {
  const { repo } = deps;


  const birthCompleted = parseBoolean(repo.getState("birth_completed"), false);


  const rawStep = repo.getState("birth_scene_step");
  const birthSceneStep: BirthSceneStep = isValidStep(rawStep)
    ? rawStep
    : "seed_idle";


  const nickname = repo.getState("nickname");
  const birthStartedAt = parseNullableNumber(repo.getState("birth_started_at"));
  const birthLastUpdatedAt = parseNullableNumber(repo.getState("birth_last_updated_at"));
  const birthFragmentGenerated = parseBoolean(repo.getState("birth_fragment_generated"), false);
  const birthResumeCount = parseNumber(repo.getState("birth_resume_count"), 0);


  const state: BirthState = {
    birthCompleted,
    birthSceneStep,
    nickname,
    birthStartedAt,
    birthLastUpdatedAt,
    birthFragmentGenerated,
    birthResumeCount
  };


  return state;
}


/**
 * 최초 birth state를 보정해서 저장한다.
 * 앱 시작 시 BOOT 이벤트 전에 한 번 호출해도 좋다.
 */
export function ensureBirthStateInitialized(deps: BirthDependencies): BirthState {
  const state = resolveBirthState(deps);
  const now = getNow(deps);


  const entries: Record<string, string> = {};


  if (!deps.repo.getState("birth_completed")) entries["birth_completed"] = "false";
  if (!deps.repo.getState("birth_scene_step")) entries["birth_scene_step"] = "seed_idle";
  if (!deps.repo.getState("birth_fragment_generated")) entries["birth_fragment_generated"] = "false";
  if (!deps.repo.getState("birth_resume_count")) entries["birth_resume_count"] = "0";
  if (!deps.repo.getState("birth_started_at")) entries["birth_started_at"] = String(now);
  if (!deps.repo.getState("birth_last_updated_at")) entries["birth_last_updated_at"] = String(now);


  if (Object.keys(entries).length > 0) {
    deps.repo.setStates(entries);
    logInfo(deps, "birth_initialized", { entries });
  }


  return resolveBirthState(deps);
}


/**
 * 현재 state를 한 번에 저장
 */
function persistBirthState(deps: BirthDependencies, state: BirthState): void {
  const now = getNow(deps);


  deps.repo.setStates({
    birth_completed: String(state.birthCompleted),
    birth_scene_step: state.birthSceneStep,
    nickname: state.nickname ?? "",
    birth_started_at: state.birthStartedAt ? String(state.birthStartedAt) : "",
    birth_last_updated_at: String(now),
    birth_fragment_generated: String(state.birthFragmentGenerated),
    birth_resume_count: String(state.birthResumeCount)
  });
}


/**
 * BOOT 시 탄생 모드 재진입 처리
 */
export function onBirthBoot(deps: BirthDependencies): BirthTransitionResult {
  const current = ensureBirthStateInitialized(deps);
  const now = getNow(deps);


  if (current.birthCompleted) {
    return {
      state: current,
      nextStep: "birth_done",
      shouldPersist: false,
      shouldCallModel: false,
      bubbleText: null,
      debugReason: "birth already completed"
    };
  }


  const resumed: BirthState = {
    ...current,
    birthStartedAt: current.birthStartedAt ?? now,
    birthResumeCount: current.birthResumeCount + 1
  };


  persistBirthState(deps, resumed);


  logInfo(deps, "birth_resumed", {
    step: resumed.birthSceneStep,
    resumeCount: resumed.birthResumeCount
  });


  return {
    state: resumed,
    nextStep: resumed.birthSceneStep,
    shouldPersist: true,
    shouldCallModel: false,
    bubbleText: null,
    debugReason: "resume current birth step"
  };
}


/**
 * first look / nickname confirm 등 스크립트성 버블 텍스트를 제공
 */
function getScriptedBubbleForStep(step: BirthSceneStep, nickname: string | null): string | null {
  switch (step) {
    case "first_look":
      return "oh.";
    case "ask_nickname":
      return "name?";
    case "nickname_confirm":
      return nickname ? `${nickname}.` : null;
    default:
      return null;
  }
}


/**
 * 단순 단계 전이
 */
function getNextLinearStep(step: BirthSceneStep): BirthSceneStep {
  switch (step) {
    case "seed_idle":
      return "seed_stir";
    case "seed_stir":
      return "sprout_emerge";
    case "sprout_emerge":
      return "first_look";
    case "first_look":
      return "ask_nickname";
    case "ask_nickname":
      return "nickname_confirm";
    case "nickname_confirm":
      return "first_fragment";
    case "first_fragment":
      return "birth_done";
    case "birth_done":
      return "birth_done";
    default:
      return "seed_idle";
  }
}


/**
 * D+0 단계 전이
 * renderer에서 이벤트를 main으로 보내면 이 함수가 state를 결정한다.
 */
export function advanceBirthStep(
  deps: BirthDependencies,
  event: BirthEvent
): BirthTransitionResult {
  const current = resolveBirthState(deps);


  if (current.birthCompleted && event.type !== "FORCE_COMPLETE") {
    return {
      state: current,
      nextStep: "birth_done",
      shouldPersist: false,
      shouldCallModel: false,
      bubbleText: null,
      debugReason: "birth already completed"
    };
  }


  const nextState: BirthState = { ...current };


  switch (event.type) {
    case "BOOT":
      return onBirthBoot(deps);


    case "ANIMATION_DONE": {
      if (current.birthSceneStep === "ask_nickname") {
        return {
          state: current,
          nextStep: "ask_nickname",
          shouldPersist: false,
          shouldCallModel: false,
          bubbleText: "name?",
          debugReason: "waiting for nickname input"
        };
      }


      const nextStep = getNextLinearStep(current.birthSceneStep);
      nextState.birthSceneStep = nextStep;


      persistBirthState(deps, nextState);
      logInfo(deps, "birth_step_completed", {
        previousStep: current.birthSceneStep,
        nextStep
      });


      return {
        state: nextState,
        nextStep,
        shouldPersist: true,
        shouldCallModel: nextStep === "first_fragment",
        bubbleText: getScriptedBubbleForStep(nextStep, nextState.nickname),
        debugReason: "animation step advanced"
      };
    }


    case "NICKNAME_SUBMITTED": {
      if (current.birthSceneStep !== "ask_nickname") {
        return {
          state: current,
          nextStep: current.birthSceneStep,
          shouldPersist: false,
          shouldCallModel: false,
          bubbleText: null,
          debugReason: "nickname ignored because current step is not ask_nickname"
        };
      }


      const sanitized = sanitizeNickname(event.nickname);
      if (!sanitized) {
        return {
          state: current,
          nextStep: "ask_nickname",
          shouldPersist: false,
          shouldCallModel: false,
          bubbleText: "name?",
          debugReason: "invalid nickname input"
        };
      }


      nextState.nickname = sanitized;
      nextState.birthSceneStep = "nickname_confirm";


      persistBirthState(deps, nextState);


      logInfo(deps, "birth_nickname_saved", {
        nickname: sanitized
      });


      return {
        state: nextState,
        nextStep: "nickname_confirm",
        shouldPersist: true,
        shouldCallModel: false,
        bubbleText: sanitized,
        debugReason: "nickname accepted"
      };
    }


    case "FIRST_FRAGMENT_REQUEST": {
      if (current.birthSceneStep !== "first_fragment") {
        return {
          state: current,
          nextStep: current.birthSceneStep,
          shouldPersist: false,
          shouldCallModel: false,
          bubbleText: null,
          debugReason: "first fragment request ignored because current step is not first_fragment"
        };
      }


      return {
        state: current,
        nextStep: "first_fragment",
        shouldPersist: false,
        shouldCallModel: true,
        bubbleText: null,
        debugReason: "ready to call model for first fragment"
      };
    }


    case "FIRST_FRAGMENT_DONE": {
      if (current.birthSceneStep !== "first_fragment") {
        return {
          state: current,
          nextStep: current.birthSceneStep,
          shouldPersist: false,
          shouldCallModel: false,
          bubbleText: null,
          debugReason: "first fragment done ignored because current step is not first_fragment"
        };
      }


      nextState.birthFragmentGenerated = true;
      nextState.birthSceneStep = "birth_done";


      persistBirthState(deps, nextState);


      logInfo(deps, "birth_fragment_generated", {
        fragment: event.fragment,
        usedFallback: event.usedFallback
      });


      return {
        state: nextState,
        nextStep: "birth_done",
        shouldPersist: true,
        shouldCallModel: false,
        bubbleText: event.fragment,
        usedFallback: event.usedFallback,
        debugReason: "first fragment completed"
      };
    }


    case "FORCE_COMPLETE": {
      return completeBirth(deps);
    }


    default: {
      const exhaustiveCheck: never = event;
      throw new Error(`Unhandled birth event: ${JSON.stringify(exhaustiveCheck)}`);
    }
  }
}


/**
 * 모델 호출 + fallback 처리
 * current step이 first_fragment일 때만 사용
 */
export async function generateBirthFirstFragment(
  deps: BirthDependencies
): Promise<{ fragment: string; usedFallback: boolean }> {
  const state = resolveBirthState(deps);


  if (state.birthSceneStep !== "first_fragment") {
    throw new Error("generateBirthFirstFragment: current step is not first_fragment.");
  }


  try {
    if (!deps.generateFirstFragment) {
      throw new Error("generateFirstFragment dependency is missing.");
    }


    const raw = await deps.generateFirstFragment(state.nickname);
    const fragment = sanitizeBirthFragment(raw);


    if (!fragment) {
      throw new Error("Generated fragment was empty or invalid.");
    }


    return { fragment, usedFallback: false };
  } catch (error) {
    logWarn(deps, "birth_fragment_fallback_used", {
      error: error instanceof Error ? error.message : String(error)
    });


    const fallback = pickRandomFallbackFragment();
    return { fragment: fallback, usedFallback: true };
  }
}


/**
 * birth fragment는 1~5단어 수준의 짧은 조각만 허용
 */
export function sanitizeBirthFragment(raw: string): string | null {
  const trimmed = raw.trim().replace(/\s+/g, " ");
  if (!trimmed) return null;


  // 너무 길면 앞 5단어만 사용
  const words = trimmed.split(" ").slice(0, 5);
  const joined = words.join(" ").trim();


  if (!joined) return null;
  return joined;
}


function pickRandomFallbackFragment(): string {
  const index = Math.floor(Math.random() * DEFAULT_FIRST_FRAGMENT_FALLBACKS.length);
  return DEFAULT_FIRST_FRAGMENT_FALLBACKS[index];
}


/**
 * 탄생 완료 처리
 * 일반 부팅으로 넘기기 전에 반드시 호출
 */
export function completeBirth(deps: BirthDependencies): BirthTransitionResult {
  const current = resolveBirthState(deps);
  const now = getNow(deps);


  const completed: BirthState = {
    ...current,
    birthCompleted: true,
    birthSceneStep: "birth_done",
    birthStartedAt: current.birthStartedAt ?? now
  };


  persistBirthState(deps, completed);


  deps.repo.setStates({
    birth_completed: "true",
    birth_scene_step: "birth_done",
    last_seen_at: String(now)
  });


  // 첫 snapshot 저장
  try {
    const snapshot = deps.createInitialSnapshot
      ? deps.createInitialSnapshot(completed.nickname)
      : createDefaultInitialSnapshot(now);


    deps.repo.setState("last_snapshot_json", JSON.stringify(snapshot));
  } catch (error) {
    logError(deps, "birth_initial_snapshot_failed", {
      error: error instanceof Error ? error.message : String(error)
    });
  }


  logInfo(deps, "birth_completed", {
    nickname: completed.nickname
  });


  return {
    state: completed,
    nextStep: "birth_done",
    shouldPersist: true,
    shouldCallModel: false,
    bubbleText: null,
    debugReason: "birth completed"
  };
}

    visible_item: null,

    energy_hint: 0.42,

  };

// src/renderer/lifecycle/BirthSceneController.ts
// Minari D+0 birth scene controller
// PixiJS renderer 쪽에서 탄생 연출을 관리한다.


import { Container, Graphics, Text, TextStyle, FederatedPointerEvent } from "pixi.js";


type BirthSceneStep =
  | "seed_idle"
  | "seed_stir"
  | "sprout_emerge"
  | "first_look"
  | "ask_nickname"
  | "nickname_confirm"
  | "first_fragment"
  | "birth_done";


interface BirthState {
  birthCompleted: boolean;
  birthSceneStep: BirthSceneStep;
  nickname: string | null;
  birthStartedAt: number | null;
  birthLastUpdatedAt: number | null;
  birthFragmentGenerated: boolean;
  birthResumeCount: number;
}


interface BirthTransitionResult {
  state: BirthState;
  nextStep: BirthSceneStep;
  shouldPersist: boolean;
  shouldCallModel: boolean;
  bubbleText: string | null;
  debugReason: string;
  usedFallback?: boolean;
}


interface BirthApi {
  getState(): Promise<BirthState>;
  advance(event: unknown): Promise<BirthTransitionResult>;
  firstFragment(): Promise<BirthTransitionResult>;
}


interface BubbleLike {
  show(text: string): void;
  hide(): void;
}


interface SproutLike extends Container {
  setScale?(value: number): void;
  playNudge?(): void;
  setMood?(mood: string): void;
  setVisibleItem?(item: string | null): void;
}


interface BirthSceneControllerOptions {
  root: Container;
  sprout: SproutLike;
  bubble: BubbleLike;
  birthApi: BirthApi;
  onComplete?: () => void;
}


interface NicknameOverlayRefs {
  container: HTMLDivElement;
  input: HTMLInputElement;
  button: HTMLButtonElement;
  hint: HTMLDivElement;
}


export class BirthSceneController {
  private readonly root: Container;
  private readonly sprout: SproutLike;
  private readonly bubble: BubbleLike;
  private readonly birthApi: BirthApi;
  private readonly onComplete?: () => void;


  private isRunning = false;
  private isDisposed = false;
  private currentStep: BirthSceneStep | null = null;
  private nicknameOverlay: NicknameOverlayRefs | null = null;
  private overlayHost: HTMLDivElement | null = null;


  // D+0용 아주 작은 seed 그래픽
  private seedGraphic: Graphics | null = null;


  private readonly bubbleTextStyle = new TextStyle({
    fill: 0x2f241c,
    fontSize: 18,
    fontFamily: "Arial",
    fontWeight: "600"
  });


  constructor(options: BirthSceneControllerOptions) {
    this.root = options.root;
    this.sprout = options.sprout;
    this.bubble = options.bubble;
    this.birthApi = options.birthApi;
    this.onComplete = options.onComplete;
  }


  /**
   * D+0 씬 시작
   */
  async start(): Promise<void> {
    if (this.isRunning || this.isDisposed) return;


    this.isRunning = true;


    try {
      const state = await this.birthApi.getState();


      if (state.birthCompleted) {
        this.finish();
        return;
      }


      await this.runStep(state.birthSceneStep, state);
    } catch (error) {
      console.error("[BirthSceneController] start failed:", error);
      // 극단 fallback: 탄생 실패 시 그냥 종료시키지 말고, 최소한 말풍선 하나 보여주고 끝
      this.safeShowBubble("oh.");
    }
  }


  /**
   * 정리
   */
  dispose(): void {
    this.isDisposed = true;
    this.isRunning = false;
    this.cleanupSeedGraphic();
    this.removeNicknameOverlay();
  }


  /**
   * step별 실행기
   */
  private async runStep(step: BirthSceneStep, state?: BirthState): Promise<void> {
    if (this.isDisposed) return;


    this.currentStep = step;


    switch (step) {
      case "seed_idle":
        await this.playSeedIdle();
        return;


      case "seed_stir":
        await this.playSeedStir();
        return;


      case "sprout_emerge":
        await this.playSproutEmerge();
        return;


      case "first_look":
        await this.playFirstLook();
        return;


      case "ask_nickname":
        await this.playAskNickname(state);
        return;


      case "nickname_confirm":
        await this.playNicknameConfirm(state);
        return;


      case "first_fragment":
        await this.playFirstFragment();
        return;


      case "birth_done":
        this.finish();
        return;


      default:
        console.warn("[BirthSceneController] unknown step:", step);
        this.finish();
    }
  }


  /**
   * seed_idle
   * 아주 작은 씨앗 상태를 잠깐 보여줌
   */
  private async playSeedIdle(): Promise<void> {
    this.cleanupSeedGraphic();
    this.seedGraphic = new Graphics();
    this.seedGraphic.beginFill(0x8c6a45);
    this.seedGraphic.drawEllipse(0, 0, 18, 12);
    this.seedGraphic.endFill();
    this.seedGraphic.x = this.sprout.x || 0;
    this.seedGraphic.y = this.sprout.y || 0;
    this.root.addChild(this.seedGraphic);


    this.sprout.visible = false;
    this.bubble.hide();


    await this.delay(1000);


    const result = await this.birthApi.advance({ type: "ANIMATION_DONE" });
    await this.runStep(result.nextStep, result.state);
  }


  /**
   * seed_stir
   * 씨앗이 작게 꿈틀거림
   */
  private async playSeedStir(): Promise<void> {
    if (!this.seedGraphic) {
      await this.playSeedIdle();
      return;
    }


    await this.wiggle(this.seedGraphic, 3, 240, 3);


    const result = await this.birthApi.advance({ type: "ANIMATION_DONE" });
    await this.runStep(result.nextStep, result.state);
  }


  /**
   * sprout_emerge
   * seed를 숨기고 새싹을 scale-up
   */
  private async playSproutEmerge(): Promise<void> {
    this.cleanupSeedGraphic();

    this.sprout.alpha = 1;
    this.sprout.scale.set(0.2);

    await this.runStep(result.nextStep, result.state);
  }


  /**
   * first_look
   * 첫 반응 버블 + 작은 까딱
   */
  private async playFirstLook(): Promise<void> {
    this.safeShowBubble("oh.");
    this.tryPlayNudge();


    await this.delay(900);
    this.bubble.hide();


    const result = await this.birthApi.advance({ type: "ANIMATION_DONE" });
    if (result.bubbleText) {
      this.safeShowBubble(result.bubbleText);
      await this.delay(700);
      this.bubble.hide();
    }
    await this.runStep(result.nextStep, result.state);
  }


  /**
   * ask_nickname
   * HTML overlay 입력 UI를 띄움
   */
  private async playAskNickname(state?: BirthState): Promise<void> {
    if (state?.nickname && state.nickname.trim()) {
      // 이미 nickname이 있으면 confirm으로 점프
      const next = await this.birthApi.advance({
        type: "NICKNAME_SUBMITTED",
        nickname: state.nickname
      });
      await this.runStep(next.nextStep, next.state);
      return;
    }


    this.safeShowBubble("name?");

  /**
   * nickname_confirm
   */
  private async playNicknameConfirm(state?: BirthState): Promise<void> {
    this.removeNicknameOverlay();


    const nickname = state?.nickname?.trim() || "friend";
    this.safeShowBubble(nickname);
    this.tryPlayNudge();

    const result = await this.birthApi.advance({ type: "ANIMATION_DONE" });
    await this.runStep(result.nextStep, result.state);
  }


  /**
   * first_fragment
   * 여기서만 모델 호출
   */
  private async playFirstFragment(): Promise<void> {

      return;

  private delay(ms: number): Promise<void> {
    return new Promise((resolve) => {
      window.setTimeout(resolve, ms);
    });

  /**
   * 유틸: scale 애니메이션
   */
  private async scaleTo(target: Container, targetScale: number, durationMs: number): Promise<void> {
    const start = performance.now();
    const from = target.scale.x;


    return new Promise((resolve) => {
      const tick = (now: number) => {
        const t = Math.min(1, (now - start) / durationMs);
        const eased = this.easeOutCubic(t);
        const value = from + (targetScale - from) * eased;


        target.scale.set(value);


        if (t >= 1) {
          resolve();
          return;
        }


        requestAnimationFrame(tick);
      };


      requestAnimationFrame(tick);
    });
  }


  /**
   * 유틸: 좌우 wiggle
   */
  private async wiggle(target: Container, amplitudePx: number, durationMs: number, repeat: number): Promise<void> {
    const originalX = target.x;


    for (let i = 0; i < repeat; i += 1) {
      await this.tweenX(target, originalX - amplitudePx, durationMs / 3);
      await this.tweenX(target, originalX + amplitudePx, durationMs / 3);
      await this.tweenX(target, originalX, durationMs / 3);
    }


    target.x = originalX;
  }


  private async tweenX(target: Container, to: number, durationMs: number): Promise<void> {
    const start = performance.now();
    const from = target.x;


    return new Promise((resolve) => {
      const tick = (now: number) => {
        const t = Math.min(1, (now - start) / durationMs);
        const eased = this.easeInOutSine(t);
        target.x = from + (to - from) * eased;


        if (t >= 1) {
          resolve();
          return;
        }


        requestAnimationFrame(tick);
      };


      requestAnimationFrame(tick);
    });
  }


  private easeOutCubic(t: number): number {
    return 1 - Math.pow(1 - t, 3);
  }


  private easeInOutSine(t: number): number {
    return -(Math.cos(Math.PI * t) - 1) / 2;
  }
}

      advance(event: unknown): Promise<any>;
      firstFragment(): Promise<any>;
    };
  }
}
