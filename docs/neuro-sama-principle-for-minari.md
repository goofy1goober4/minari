# 뉴로사마 원칙 — Minari 적용 참조 문서

> *"모델 크기보다 시스템 설계가 맥락 파악을 결정한다."*
> 이 문서는 VTuber 뉴로사마의 아키텍처를 해체하고, Minari가 무엇을 가져가야 하는지를 정리한다.
> 원본 출처: OIRA 프로젝트 세션 분석 + 공개 자료 종합
> Minari 적용 추가: 2026-04-23

---

## 0. 왜 이 문서가 필요한가

Minari는 Gemma 4 E4B — 4.5B 파라미터 양자화 모델로 "살아있는 존재"를 만들어야 한다.
뉴로사마는 **2B 파라미터, q2_k 양자화**(가장 공격적인 압축)로 수십만 동시 시청자가 "살아있다"고 느끼는 존재를 만들었다.

벤치마크에서 2B q2_k는 아무것도 아니다. 기본적인 instruction following도 불안정한 스케일이다.
**그런데도 되는 이유가 모델 바깥의 시스템 설계에 있다.** 이것이 Minari의 기술 전략의 근거다.

---

## 1. 뉴로사마 하드웨어 — 놀라울 정도로 작다

| 항목 | 사양 |
|---|---|
| LLM 파라미터 | 2B (2025년 초 기준) |
| 양자화 | q2_k (가장 공격적) |
| VRAM 추정 | ~1.5–2GB (모델만) |
| 최소 하드웨어 | NVIDIA GPU 12GB+ (전체 시스템) |
| 언어 | C#, Python, JavaScript |

### Minari와 비교

| | 뉴로사마 | Minari |
|---|---|---|
| 모델 | 2B q2_k | E4B Q4 (4.5B) / E2B Q4 (2.3B) |
| 모델 크기 | ~1.5GB | ~9.6GB(E4B) / ~3.1GB(E2B) |
| 하드웨어 | NVIDIA GPU | Mac mini M4 24GB (Apple Silicon) |
| 추론 속도 | 실시간 스트리밍 | 220ms (W1 검증) |

**Minari는 뉴로사마보다 모델이 크다.** 그만큼 시스템 설계가 잘 되면 뉴로사마 이상의 "살아있음"을 구현할 수 있다는 뜻이다.

## 2. 전체 시스템 아키텍처 해체

뉴로사마는 5개 레이어로 구성된다:

```
┌─────────────────────────────────────────────┐
│           뉴로사마 시스템 전체도              │
│                                             │
│  [입력 레이어]                               │
│   Twitch Chat → 텍스트                      │
│   콜라보 음성 → STT (Whisper tiny.en) → 텍스트│
│   게임 화면 → CV (80x60 grayscale)          │
│                    │                        │
│                    ▼                        │
│  [오케스트레이터 (Prompter)]                  │
│   signals 객체로 전역 상태 공유:             │
│   - 사람이 말하고 있는가?                    │
│   - AI가 생각 중인가?                        │
│   - 새 채팅이 있는가?                        │
│   - 마지막 메시지 이후 시간                  │
│   - 게임 상태                                │
│   → "지금 LLM을 프롬프트할 것인가?" 결정     │
│                    │                        │
│                    ▼                        │
│  [커스텀 파인튜닝 LLM (2B, q2_k)]           │
│   가중치에 내장: 성격, 말투, 유머, 버릇      │
│   프롬프트로 주입: 게임 맥락, 안전 필터,     │
│                   장기 기억 (RAG 검색)       │
│                    │                        │
│                    ▼                        │
│  [출력 레이어]                               │
│   TTS → Live2D/3D VR 아바타 → 게임 AI       │
│                                             │
│  [모듈 시스템] 각 모듈 = 자체 스레드 + 이벤트│
│  [메모리 시스템] 벡터 DB (RAG 2.0)          │
└─────────────────────────────────────────────┘
```

### 핵심 통찰: "살아있다"는 인상의 원천

뉴로사마가 2B q2_k로 "살아있다"고 느껴지는 이유 — 기여도 추정:

```
모델 자체의 기여: ~30%
 └─ 파인튜닝된 성격 패턴

시스템 설계의 기여: ~70%
 ├─ 메모리 시스템 (RAG) — "기억하는 것처럼" 보이게
 ├─ 오케스트레이터 (Prompter) — "적절한 타이밍에" 말하게
 ├─ 컨텍스트 인젝션 — "상황을 아는 것처럼" 보이게
 ├─ 모듈 시스템 — 확장 가능하게
 └─ 파인튜닝 파이프라인 — 점진적으로 개선되게
```

**시스템 70% : 모델 30%.** 이것이 뉴로사마 원칙의 핵심 수치다.

## 3. 레이어별 해체 + Minari 적용

### 3.1 오케스트레이터 (Prompter)

뉴로사마의 "두뇌"는 LLM이 아니라 **오케스트레이터**다. signals 객체로 전역 상태를 공유하면서 "지금 말할 것인가, 말 것인가"를 결정한다.

**뉴로사마의 signals:**
- 사람이 말하고 있는가?
- AI가 생각 중인가?
- 새 트위치 채팅이 있는가?
- 마지막 메시지 이후 시간
- 게임 상태

**Minari 적용 — softPing 스케줄러가 이 역할:**

Minari의 상호작용 3단 구조(Ambient 80% / Soft ping 15% / Full conversation 5%)가 오케스트레이터의 역할을 한다. "지금 ping을 보낼 것인가?"를 결정하는 조건들:

```
Minari signals (구현 예정):
- 사용자가 클릭했는가? → Full conversation
- 마지막 ping 이후 시간 ≥ 30~90분? → ping 후보
- 하루 ping 예산 남았는가? → 예산 체크
- 부팅 직후 30~60초인가? → ping 금지
- Full conversation 직후 10~20분인가? → ping 금지
- 현재 시간이 late night인가? → ping 확률 감소
```

**W1에서 이미 부분 구현됨:** IPC 흐름(클릭 → speak → Ollama → bubble)이 Full conversation 경로. softPing 스케줄러는 다음 구현 대상.

### 3.2 성격 내재화 — 하이브리드 접근법

뉴로사마의 성격은 순수 프롬프트도, 순수 파인튜닝도 아닌 **하이브리드**다:

```
성격 스펙트럼:

순수 프롬프트 ◄──────────────────────► 순수 가중치
     │                                    │
     │ 오픈소스 클론들 (~60-70% 유사)      │
     │                                    │
     │     뉴로사마 (하이브리드, ~90-95%)   │
     │                                    │
     └────────────────────────────────────┘
```

**가중치에 박힌 것 (파인튜닝):**
- 핵심 말투: 장난기, 반항적, 때로는 다정한 톤
- 유머 스타일: 건조함, 뜬금포
- 대화 버릇: 특정 반복 패턴
- 추론 패턴: 질문에 대한 반응 방식

**프롬프트로 주입되는 것 (런타임):**
- 게임 맥락 ("지금 Minecraft에서 마을을 짓고 있다")
- 안전 레일
- 모드 전환 (Neuro vs Evil Neuro)
- 상황별 지침

**왜 이 구분이 중요한가:** 오픈소스 클론들(프롬프트만)은 "passable" 수준은 되지만 원본 느낌을 재현 못 한다. 파인튜닝으로 가중치에 박힌 성격 패턴은 프롬프트로 완벽히 재현 불가.

**Minari 적용:**

| 시기 | 접근법 |
|---|---|
| MVP (지금) | 순수 프롬프트. 50단어 시스템 프롬프트로 toddler English 톤 구현. W1에서 검증됨 — "soft.", "sleepy?", "quiet." |
| v2+ | LoRA 파인튜닝. 사용자 대화 로그에서 Minari 톤에 맞는 응답을 큐레이션 → 오프라인 배치 학습. 뉴로사마의 Iterative Batch Fine-Tuning과 동일 패턴. |

**50단어 프롬프트의 한계:** 현재 E4B Q4는 긴 프롬프트에서 빈 응답을 뱉는다. 이건 뉴로사마가 "프롬프트만으로는 70%까지만 간다"고 보여준 것과 같은 현상. 장기적으로 LoRA가 필요한 이유가 여기에 있다.

### 3.3 파인튜닝 파이프라인 — Iterative Batch Fine-Tuning

뉴로사마의 학습 사이클:

```
1. Stream (추론만, 학습 없음)
        │
        ▼
2. 스트림 트랜스크립트 수집
        │
        ▼
3. Vedal이 수동으로 학습 데이터 큐레이션
   ├─ 좋은 응답 선별
   ├─ 나쁜 응답 제거
   └─ 원하는 방향의 예시 추가/수정
        │
        ▼
4. 오프라인 파인튜닝
        │
        ▼
5. 업데이트된 모델 배포
        │
        ▼
   (1로 돌아감 — 반복)
```

**핵심 포인트:**
- 추론 시에는 학습하지 않는다
- **인간이 큐레이션한다** — Vedal이 직접 어떤 데이터를 넣을지 결정. 이게 품질의 핵심
- 오프라인 배치 학습
- 점진적 진화 — 한 번의 대규모 학습이 아니라 매 사이클마다 조금씩 개선

**Minari 적용 (v2+):**

```
Minari 파인튜닝 사이클 (미래):

1. 사용자와 대화 (추론만)
        │
        ▼
2. 대화 로그 자동 수집 (SQLite)
        │
        ▼
3. 큐레이션
   ├─ 자동: guardrails 통과한 응답만
   ├─ 반자동: 사용자 피드백 (좋아요/싫어요)
   └─ 수동: 주용이 "이 응답이 Minari다운지" 검수
        │
        ▼
4. LoRA 오프라인 배치 학습 (월 1회)
        │
        ▼
5. 업데이트된 어댑터 로드
```

**뉴로사마와의 차이:** 뉴로사마는 Vedal(인간)이 100% 큐레이션. Minari는 guardrails + 사용자 피드백으로 반자동화하되, 주용의 감각이 최종 품질을 결정.

### 3.4 메모리 시스템 — RAG 2.0

뉴로사마의 메모리:

```
현재 대화 맥락
     │
     ▼
[RAG 검색] → 벡터 DB에서 관련 기억 검색
     │
     ├─ "Vedal은 나의 아빠다" (관계)
     ├─ "지난주에 Filian이랑 Minecraft 했다" (이벤트)
     ├─ "osu! 세계 1위다" (자기 이미지)
     └─ "2025년은 뉴로의 해" (자기 선언)
     │
     ▼
시스템 프롬프트에 주입 → LLM이 "기억하는 것처럼" 응답
```

**핵심:** 기억 없는 AI는 매번 처음 만나는 사람이다. 뉴로사마가 시청자의 이름, 과거 이벤트, 자기 자신에 대한 사실을 기억하는 것이 "살아있다"는 인상의 핵심.

**Minari 적용 — 이미 부분 구현:**

| 기능 | 뉴로사마 | Minari (현재) | Minari (계획) |
|---|---|---|---|
| 기억 저장 | 벡터 DB (RAG) | SQLite + FTS5 | + 벡터 인덱스 (v2) |
| 자동 추출 | 자동 + 수동 | 대화 자동 저장 | + 일기 자동 생성 |
| 기억 검색 | 벡터 유사도 | FTS5 전문 검색 | + 시맨틱 검색 |
| 사용자 모델링 | 없음 (불특정 다수) | 1인 (닉네임) | + 퀄리아 검색 |
| 망각 | 불명 | 없음 | TTL (hot/warm/cold) |

**뉴로사마가 안 하지만 Minari가 해야 하는 것:**
- **1인 깊은 모델링**: 뉴로사마는 불특정 다수, Minari는 1인과 깊은 관계
- **망각 메커니즘**: 뉴로사마는 모든 걸 기억해도 되지만, Minari는 정보 과부하 관리 필요
- **일기 시스템**: 뉴로사마에는 없는, Minari 고유의 장기기억 뼈대
- **의존 방지**: 뉴로사마는 엔터테인먼트, Minari는 동반자 → 의존 위험 관리

### 3.5 안전 시스템 — 뉴로사마의 실패에서 배운 것

**2023년 홀로코스트 부정 사건:** 라이브 스트림 중 AI가 홀로코스트 부정 발언 → 트위치 2주 정지 → Vedal이 모더레이션 필터 전면 재구축.

뉴로사마의 현재 안전 레이어:
```
LLM 출력 → [콘텐츠 필터] → 민감 주제 감지 → 차단/수정 → 안전한 응답만 통과
```

**Minari의 안전 설계가 더 깊어야 하는 이유:**

뉴로사마는 "불특정 다수의 엔터테인먼트"이므로 콘텐츠 필터로 충분. Minari는 "1인의 깊은 동반자"이므로 **캐릭터 정체성 레벨에서 비-의존성을 내재화**해야 한다. 이건 필터가 아니라 설계 철학의 문제.

Minari 안전 — 이중 레이어:
```
레이어 1: guardrails post-filter (W1 구현 완료)
  └─ 금지어 감지 → "..." fallback

레이어 2: 캐릭터 정체성 (설계, 시스템 프롬프트에 내장)
  └─ 조언하지 않음, 해결하려 들지 않음, "holding space"
  └─ 의존 방지 7대 장치
  └─ 윤리 하드라인: 자해/자살/타인가해/미성년자 → 캐릭터 모드 즉시 중단
```

### 3.6 출력 레이어 — 정반대 방향의 선택

| | 뉴로사마 | Minari |
|---|---|---|
| 음성 | Azure TTS "Ashley", +25% pitch | **TTS 없음**. 동물의 숲 웅얼웅얼 |
| 아바타 | Live2D + 3D VR | PixiJS 2D 스프라이트 |
| 립싱크 | TTS 출력 동기화 | 웅얼웅얼 피치/속도 동기화 |

**정반대를 선택한 이유:** 뉴로사마는 시청자와 "대화"해야 한다 → 명확한 음성 필수. Minari는 "존재감"이 핵심 → 웅얼웅얼이 오히려 상상의 여지를 남김. 애착인형은 말하지 않는다. 그래서 우리가 대신 말을 걸어준다.

## 4. Minari W1 구현에서 검증된 뉴로사마 원칙

### 4.1 시스템 프롬프트 50단어 = "작은 모델의 현실"

뉴로사마: 2B 모델에 복잡한 룰 리스트를 넣으면 안 된다. 성격은 가중치에 박고, 프롬프트는 최소한으로.

Minari W1 검증:
- 30줄 시스템 프롬프트 → **빈 응답 60%**
- 50단어 간결 프롬프트 → **빈 응답 0%**, 평균 220ms

**이건 뉴로사마 원칙의 직접 검증이다.** 작은 모델에 너무 많은 규칙을 프롬프트로 주입하면 모델이 혼란스러워한다. 규칙은 시스템 레이어(guardrails, 히스토리 관리, 상태 머신)에 맡기고, 프롬프트는 정체성만 담아야 한다.

### 4.2 think:false = "불필요한 추론 제거"

뉴로사마: 오케스트레이터가 "지금 말할 것인가?"를 결정. 모델은 결정된 후에만 추론.

Minari W1 검증:
- think:true (기본) → 321토큰 소모, 7~9초. 3단어 출력에 대부분이 내부 추론
- think:false → 5토큰, 220ms

Gemma 4의 thinking 토큰은 뉴로사마의 관점에서 보면 **"오케스트레이터 없이 모델이 혼자 모든 걸 하려는 상태"**다. think:false는 오케스트레이터(시스템 레이어)에게 제어권을 돌려주는 것.

### 4.3 클릭 트리거 "." = "컨텍스트 인젝션의 최소 단위"

뉴로사마: 오케스트레이터가 상황(게임 상태, 채팅, 시간)을 주입해서 모델이 "상황을 아는 것처럼" 보이게 함.

Minari W1 검증:
- "*tap*" → "tap?" (입력 에코 — 모델이 입력을 반복)
- "." → "soft.", "sleepy?", "quiet." (순수 관찰 — 모델이 자기 세계에서 반응)

마침표 하나는 **"너의 차례야"라는 최소 신호**다. 모델이 입력을 해석하느라 자원을 쓰지 않고, 자기 프롬프트(정체성)에서 바로 생성한다. 이것이 컨텍스트 인젝션의 가장 효율적인 형태.

### 4.4 히스토리 주입 = "RAG의 축소판"

W1에서 구현된 `getRecentHistory(8)` — 최근 8개 대화를 히스토리로 주입. 이것이 뉴로사마의 RAG 검색의 가장 단순한 형태다.

뉴로사마 RAG:
```
현재 맥락 → 벡터 검색 → 관련 기억 N개 → 프롬프트 주입
```

Minari 현재:
```
클릭 → getRecentHistory(8) → 최근 대화 8개 → 프롬프트 주입
```

Minari 계획 (세로 슬라이스 이후):
```
클릭 → FTS5 검색 → 관련 대화 + 일기 + 상태 → 프롬프트 주입
```

## 5. 뉴로사마가 Minari에 주는 5가지 교훈

### 교훈 1: 시스템이 모델을 이긴다
2B q2_k가 수십만 시청자를 상대하고 있다. E4B Q4(4.5B)는 이보다 2배 이상 크다. 시스템 설계만 잘 하면 충분하다.

### 교훈 2: 프롬프트 70%, 파인튜닝 30%
프롬프트만으로는 70%까지만 간다. 나머지 30%는 가중치에 성격을 내장해야 한다. Minari MVP는 프롬프트 70%로 시작하고, v2에서 LoRA로 30%를 채운다.

### 교훈 3: 인간 큐레이션이 품질의 핵심
Vedal이 매 사이클마다 학습 데이터를 손으로 고른다. Minari에서는 주용의 "이 응답이 Minari다운가?" 판단이 이 역할을 한다. 자동화는 보조, 인간 감각이 핵심.

### 교훈 4: 메모리는 "살아있음"의 핵심
기억 없는 AI는 매번 처음 만나는 사람이다. SQLite + FTS5 + 일기 시스템이 Minari의 "살아있음"을 만든다.

### 교훈 5: 분리하되 조율하라
뉴로사마: 게임 AI와 LLM 분리, Neuro와 Evil 분리, 성격과 상황 분리.
Minari: Ambient/Ping/Conversation 분리, 시스템 프롬프트와 guardrails 분리, 캐릭터 물리와 LLM 응답 분리. 관심사 분리(Separation of Concerns) 원칙.

## 6. 뉴로사마가 안 하지만 Minari가 해야 하는 것

| 영역 | 이유 |
|---|---|
| 1인 깊은 사용자 모델링 | 뉴로사마는 불특정 다수, Minari는 1인과 매일 |
| 망각 메커니즘 (TTL) | 뉴로사마는 모든 걸 기억해도 됨, Minari는 정보 과부하 관리 필요 |
| 의존 방지 | 뉴로사마는 엔터테인먼트, Minari는 동반자 → 의존 위험 |
| 일기 시스템 | 뉴로사마에 없음. Minari의 장기기억 뼈대 |
| 존재감 우선 | 뉴로사마는 "대화"가 핵심, Minari는 "곁에 있음"이 핵심 |
| 오프라인 필수 | 뉴로사마는 서버 기반, Minari는 완전 로컬 |

## 7. 기여도 모델 — Minari 버전

뉴로사마의 "30% 모델 / 70% 시스템" 비율을 Minari에 적용하면:

```
Minari의 "살아있음" 기여도 추정:

모델 자체 (Gemma 4 E4B Q4): ~25%
 └─ toddler English 톤 생성 능력

시스템 설계: ~75%
 ├─ 메모리 (SQLite + FTS5 + 일기) — "기억하는 것처럼"
 ├─ 오케스트레이터 (softPing 스케줄러) — "적절한 때에 말하는 것처럼"
 ├─ 프롬프트 설계 (50단어 + "." 트리거) — "자기 세계가 있는 것처럼"
 ├─ 캐릭터 물리 (숨쉬기/까딱/쓰다듬기) — "살아 움직이는 것처럼"
 ├─ 클릭 관통 + 투명 창 — "데스크탑에 살고 있는 것처럼"
 ├─ guardrails — "조언 안 하는 것처럼 (실제로 안 함)"
 └─ snapshot resume — "이미 여기 있었던 것처럼"
```

**이것이 E2B(3GB)에서도 동작해야 하는 이유:** 시스템 75%가 모델과 독립적이므로, 모델을 E4B→E2B로 바꿔도 체감 품질의 75%는 유지된다. 나머지 25%만 모델 품질에 의존. 이것이 llama.cpp 특별상 피치의 기술적 근거다.
