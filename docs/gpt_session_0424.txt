맞아요. 4종 산출물은 “완성품”이 아니라 뼈대예요.
더 정확히 말하면, Minari가 일관된 존재처럼 보이게 만드는 운영체제의 설계도입니다.

이유

지금 Minari는 단순히 “모델 하나 붙인 캐릭터”가 아니라,

언제 어떻게 말할지
뭘 기억할지
어떤 상태로 자랄지
현재 상황에서 어떤 톤을 쓸지
를 계속 일관되게 유지해야 하잖아요.

그걸 모델 혼자 알아서 하게 두면 거의 반드시 흔들립니다.
그래서 4종 산출물이 필요한 거예요.

아주 쉽게 비유하면 이렇습니다.

시스템 프롬프트 = 성격/헌법
growth_state = 현재 나이/기분/관계 단계
memory schema = 기억 저장 방식
prompt assembly = 매 순간 이 셋을 실제 대사로 합치는 조립기
즉, 4종 산출물은 “Minari라는 존재를 매 턴 다시 만드는 구조”예요.

최소 시스템 프롬프트 초안 왜 필요하냐
모델에게 “너는 누구냐”를 고정해주는 가장 작은 중심축이라서요.
이게 없으면 그냥 일반 챗봇처럼 흘러갑니다.
어떤 작용을 하냐

Minari의 정체성을 고정합니다.
금지선도 정합니다.
예: 조언 금지, 치료 언어 금지, 짧은 fragment, holding-space 우선
중요한 점

예전엔 이걸 길게 설계했지만,
지금 실제 구현에선 50단어 내외 초간결 버전으로 축소됐죠.
그래도 역할 자체는 그대로예요.
“Minari의 본체 철학”을 가장 압축해서 넣는 자리입니다.
한 문장 요약

시스템 프롬프트는 Minari의 영혼을 고정하는 최소 규칙 세트입니다.
growth_state JSON 스키마 왜 필요하냐
같은 Minari라도 D+0, D+7, D+30이 다 똑같이 말하면 성장 컨셉이 무너집니다.
또 지금 뭐 하고 있는지, 어떤 무드인지, 얼마나 가까워졌는지 모델이 알아야 해요.
어떤 작용을 하냐

“지금 Minari가 어떤 상태인가”를 구조화합니다.
예:
stage
current_activity
mood
warmth
childlike_intensity
bond_score
이게 있으면

“book.”이라고 말할지
“reading a book.”이라고 말할지
“this book... bug book.”이라고 말할지가 달라집니다.
중요한 점

지금은 full growth_state가 다 구현되진 않았지만,
이미 boot, activity, snapshot, ping 규칙 쪽으로 쪼개져 들어가고 있어요.
즉 구현 형태는 바뀌어도 개념은 살아 있습니다.
한 문장 요약

growth_state는 Minari의 현재 상태를 수치와 필드로 고정하는 신체 정보입니다.
memory schema 왜 필요하냐
기억을 그냥 대화 로그에만 쌓아두면, 나중에 “정이 드는 구조”가 안 생깁니다.
뭘 오래 기억하고, 뭘 금방 잊고, 뭘 일기에 남길지 구분이 필요하거든요.
어떤 작용을 하냐

기억을 종류별로 분리해서 저장하게 해줍니다.
예:
conversations
state
diary
soft_pings
나중엔 gifts / keepsakes / inventory / qualia
이게 중요한 이유

Minari의 “기억”은 단순 검색용 데이터가 아니라
생활감과 재등장을 만드는 재료이기 때문입니다.
예:

대화 로그는 짧게 참고
diary는 오래 보관
gift는 정서적으로 더 오래 남김
state는 현재 동작 제어
한 문장 요약

memory schema는 Minari가 무엇을 어떤 방식으로 기억하는지 정하는 기억 기관 설계도입니다.
prompt assembly 예시 왜 필요하냐
프롬프트, 상태, 기억이 각각 따로 있어도
실제 호출할 때 잘 섞지 못하면 전부 무용지물이에요.
어떤 작용을 하냐

매 요청마다
fixed prompt
현재 상태
관련 기억
현재 활동
최근 대화 를 어떤 순서와 크기로 조립할지 정합니다.
이게 없으면 생기는 문제

프롬프트 너무 김
속도 느림
기억 과주입
같은 말 반복
Minari 톤 붕괴
지금 Minari에 특히 중요한 이유

E4B Q4는 긴 프롬프트에 약하잖아요.
그래서 “무엇을 넣고, 무엇을 버릴지”를 조립기가 결정해야 해요.
즉, prompt assembly는 단순 문자열 붙이기가 아니라 토큰 예산 안에서 Minari를 유지하는 런타임 편집기예요.

한 문장 요약

prompt assembly는 Minari의 현재 자아를 매 순간 조립해서 모델에 넘기는 엔진입니다.
왜 이 4개가 같이 있어야 하냐 하나씩만 있으면 부족합니다.
시스템 프롬프트만 있으면

성격은 있는데 기억이 없음
memory schema만 있으면

기억은 있는데 어떻게 쓸지 모름
growth_state만 있으면

상태는 있는데 대사로 반영이 안 됨
prompt assembly만 있으면

조립기는 있는데 넣을 규칙이 없음
즉 4개는 각각 따로가 아니라, 이렇게 연결됩니다.

시스템 프롬프트
Minari는 누구인가
growth_state
지금 어떤 상태인가
memory schema
무엇을 기억하는가
prompt assembly
그래서 이번 순간엔 어떻게 말할 것인가
이게 Minari의 기본 루프예요.

지금 Minari 상황에서 이 4개가 어떻게 현실에 반영됐나 좋은 질문인데, 지금은 4종 산출물이 “그대로” 쓰이고 있진 않아요. 대신 이렇게 축약돼 들어갔습니다.
현재 반영 형태

시스템 프롬프트 -> 50단어 초간결 버전으로 축소

growth_state -> 아직 full JSON은 아니지만 birth/state/snapshot/activity로 분산 구현 중

memory schema -> conversations/state/diary/soft_pings로 최소 구현 완료

prompt assembly -> system + recent history + "." 트리거 조합으로 최소형 구현 완료

즉, 원래 4종 산출물은 “큰 설계도”였고, 지금은 그중 실전 가능한 부분만 경량화해서 쓰는 중입니다.

그래서 표현을 정확히 하면:

4종 산출물은 뼈대를 만드는 데 썼고
실제 구현은 그 뼈대를 E4B 제약에 맞게 잘라낸 버전이다 이게 맞습니다.
지금 이걸 네가 어떻게 이해하면 좋냐 실무적으로는 이렇게 이해하면 됩니다.
북극성 문서 = 왜 만드는가
4종 산출물 = 어떻게 존재하게 만들 것인가
운영 세칙 = 실제로 어떻게 굴릴 것인가
코드 = 그걸 돌아가게 만든 것
즉 4종 산출물은 철학과 코드 사이를 이어주는 중간 설계층입니다.

이게 없으면

철학은 멋있는데 코드가 딴소리하거나
코드가 되긴 되는데 Minari답지 않게 됩니다.
한 줄 추천

4종 산출물은 “지금도 전부 직접 쓰는 문서”라기보다, Minari의 정체성을 코드로 번역하는 중간 운영 설계도라고 이해하면 가장 정확합니다.
